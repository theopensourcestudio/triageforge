# Adoption guide

A new open-source repository should not claim adoption it does not have. Build it transparently.

1. Publish a signed `v0.1.0` release and install the action on this repository first.
2. Ask maintainers of three small public repositories to test read-only reports.
3. Open public issues for every false positive and document fixes in the changelog.
4. Publish an evaluation dataset containing synthetic or permission-cleared examples.
5. Add screenshots of real job summaries and measured before/after triage time.
6. Release frequently enough to show maintenance, without manufacturing meaningless commits.
7. Invite external contributors through clearly scoped `good first issue` tasks.

Useful metrics include unique repositories using the action, package downloads, release downloads, external issues, merged community pull requests, response time, and measured precision. Never invent or purchase stars, downloads, contributors, or usage.
